This guide covers deploying Authstack on a Dokploy-managed server. The same shape applies to any Docker-capable host.

## Prerequisites

- A running PostgreSQL instance, either Dokploy-managed or external.
- Docker Hub credentials for pulling the image.
- A domain or subdomain pointing to your server, for example `auth.yourdomain.com`.

## Docker image

The `docker-build-push` GitHub Actions workflow builds the image on every push to `main` and pushes it to Docker Hub tagged as:

```txt
<DOCKER_USERNAME>/<DOCKER_REPOSITORY>:authstack-latest
```

Set these repository secrets in GitHub before the first push:

| Secret | Description |
| --- | --- |
| `DOCKER_USERNAME` | Your Docker Hub username. |
| `DOCKER_PASSWORD` | Your Docker Hub access token. |
| `DOCKER_REPOSITORY` | Repository name, for example `brisk`. |

## Generate a key pair

Run this once on any machine with `openssl` installed:

```bash
make keys
```

Copy the two output lines and paste them as environment variables in Dokploy.

`JWT_PRIVATE_KEY` must be kept secret. `JWT_PUBLIC_KEY` can be distributed to services that verify tokens locally.

## Dokploy setup

Create a new Dokploy Application, set the source to Docker Image, and enter:

```txt
<DOCKER_USERNAME>/<DOCKER_REPOSITORY>:authstack-latest
```

Set the internal port to `8080`.

Add these environment variables in Dokploy:

| Variable | Example / Notes |
| --- | --- |
| `DATABASE_URL` | `postgres://user:pass@host:5432/authstack` |
| `JWT_PRIVATE_KEY` | Base64-encoded PKCS#8 PEM from `make keys`. |
| `JWT_PUBLIC_KEY` | Base64-encoded SPKI PEM from `make keys`. |
| `ACCESS_TOKEN_EXPIRY_SECS` | Optional. Default `900`. |
| `REFRESH_TOKEN_EXPIRY_SECS` | Optional. Default `2592000`. |
| `PORT` | Optional. Default `8080`. |
| `RUST_LOG` | Optional. Use `authstack=info` for production. |

For the **first deploy only**, also set bootstrap variables so the entrypoint can create the first instance admin:

| Variable | Example / Notes |
| --- | --- |
| `AUTHSTACK_BOOTSTRAP_EMAIL` | `admin@yourdomain.com` |
| `AUTHSTACK_BOOTSTRAP_PASSWORD` | A strong password. Remove after first successful deploy. |

The key values generated by `make keys` are single-line base64 values. Paste them directly into Dokploy without quotes.

## Health check

Configure Dokploy's health check to hit:

```txt
GET /.well-known/jwks.json
```

This returns `200` when the service is up and JWT keys are loaded.

## First deploy

Click Deploy in Dokploy. On first boot, Authstack automatically runs database migrations and bootstraps the first instance admin when bootstrap variables are set.

Check the container logs for:

```txt
bootstrap: created first instance admin
authstack listening on 0.0.0.0:8080
```

If you see `failed to initialise JWT service`, the `JWT_PRIVATE_KEY` env var is missing or malformed. Re-run `make keys` and paste the output again.

On subsequent deploys, bootstrap is skipped when an admin already exists:

```txt
bootstrap: instance admin already exists
```

## Create the first admin manually

If you prefer not to use bootstrap environment variables, run the CLI against your database before or after starting the API:

```bash
docker run --rm \
  -e DATABASE_URL='postgres://user:pass@host:5432/authstack' \
  -e AUTHSTACK_BOOTSTRAP_EMAIL='admin@yourdomain.com' \
  -e AUTHSTACK_BOOTSTRAP_PASSWORD='your-strong-password' \
  <DOCKER_USERNAME>/<DOCKER_REPOSITORY>:authstack-latest \
  bootstrap-admin
```

Then open:

```txt
https://auth.yourdomain.com/admin/login
```

## Register the first application

In the admin panel, click New application, enter a name, and create it. The app `id` and `client_secret` are shown once, so copy them immediately.

Alternatively, use the JSON API:

```bash
curl -c cookies.txt -X POST https://auth.yourdomain.com/admin/login \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "email=admin@yourdomain.com&password=your-strong-password"

curl -b cookies.txt -X POST https://auth.yourdomain.com/admin/applications \
  -H "Content-Type: application/json" \
  -d '{"name": "my-app"}'
```

```json
{
  "id": "app_01j4hz0r3fexwpbgm41z1w57at",
  "client_secret": "secret_xyz...",
  "name": "my-app"
}
```

Give the `id` and `client_secret` to your application's backend-for-frontend. It authenticates every Authstack request using HTTP Basic auth.

## Updating

Push to `main`. The `docker-build-push` workflow rebuilds and pushes `authstack-latest`. In Dokploy, trigger a redeploy or enable auto-deploy on image update.

## Troubleshooting

### Container starts and immediately stops with no error shown

This is usually a missing required environment variable. Open the container logs immediately after a failed deploy. You should see a message like:

```txt
Error: missing required env var: DATABASE_URL
```

Check that `DATABASE_URL`, `JWT_PRIVATE_KEY`, and `JWT_PUBLIC_KEY` are set and non-empty.

### JWT service fails to initialize

The private key is missing, empty, or not valid PKCS#8 format. Re-run `make keys` and paste the full base64 value from the `JWT_PRIVATE_KEY=...` line into Dokploy. Do not add quotes.

### Database connection or migration fails

- Check that `DATABASE_URL` has the correct host, port, user, password, and database name.
- If using a managed database, append `?sslmode=require` to the connection string.
- Ensure the database user can create tables.

### Cannot log in to the admin panel

- Confirm bootstrap ran successfully in the container logs.
- If the database was restored from a backup, an admin may already exist — use those credentials or reset the `admin_user` table in a controlled maintenance window.

### Consuming services reject JWTs

The consuming service is likely using the wrong public key. Copy `JWT_PUBLIC_KEY` from Dokploy and set the same value in the consuming service.

## Security checklist

- `AUTHSTACK_BOOTSTRAP_PASSWORD` is strong and removed from the runtime environment after first deploy.
- `JWT_PRIVATE_KEY` is stored only in environment variables.
- PostgreSQL is not publicly accessible.
- HTTPS is terminated at the reverse proxy.
- `client_secret` values are stored server-side only.
- Application `id` and `client_secret` are treated as credentials.
