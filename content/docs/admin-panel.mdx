---
title: Admin Panel
description: Manage applications, operators, tenant users, and organizations through the instance admin UI.
---

The admin panel at `/admin/login` is for **Authstack instance operators** only. Tenant end users never use this UI.

Operator accounts live in the `admin` schema; tenant users and organizations live in the `tenant` schema. See [Directories](/docs/directories#database-layout) for how identity is partitioned.

The UI is server-rendered with **Askama**, styled with **Tailwind CSS** and **shadcn** design tokens, and enhanced with **Datastar** for live search and inline interactions.

## Operator roles

| Role | Access |
| --- | --- |
| **Instance admin** | Full instance access: all applications, operator management, application deletion |
| **Directory admin** | Scoped to assigned directories via `admin_directory_grant`: all apps in those directories, can create apps/operators within scope |
| **App admin** | Scoped to assigned applications via `admin_app_grant`: view app details, list users, create tenant users, manage organizations |

The first instance admin is created with [`authstack bootstrap-admin`](/docs/getting-started#bootstrap-the-first-instance-admin).

Additional operators are created by an instance admin or directory admin from **Operators → Add operator**.

- **Instance admin** can create any role and assign any directory or application.
- **Directory admin** can create **app admin** or **directory admin** operators only, with grants limited to directories they control (applications must belong to those directories).

## Directories

Instance admins can manage identity boundaries from **Directories**:

- List all directories (name, slug, policy, application count, admin count)
- Open a directory to view its metadata and **directory administrators**
- **Add directory admin** from a directory's detail page (email + password) — grants `directory_admin` for that directory only
- **New directory** — set display name and slug. Users in a directory share identity across all apps in that directory; create a separate directory when apps must not share users.
- When creating an application, choose which directory it belongs to (defaults to `default`)

Directory admins see only directories they are granted in **Directories**. They can open those directories and add more directory admins (or use **Operators → Add operator**), but cannot create new directories.

## Applications

From the dashboard, instance admins can create applications and view one-time `client_secret` credentials.

All operators with access to an application can open its detail page to:

- View app metadata and user count
- Open **Users** to list tenant users (with live search)
- **Invite user** to generate a shareable link (user sets their own password)
- **Add user** to provision a tenant account directly (name, email, password)
- Open **Permissions** to define the app's permission catalog
- Open **Organizations** to browse orgs, manage roles, and manage members

Instance admins can also delete an application from the app detail page. This removes all tenant data for that application.

## Permissions

From an application's **Permissions** page you can define permission keys (e.g. `invoice:write`, `org:invite`) that your app checks in JWT claims or your backend. New permissions are automatically granted to existing **owner** roles in all organizations for that application.

See [Permissions and roles](/docs/permissions).

## Organizations

From an application's **Organizations** page you can:

- List all organizations for the application
- **Create an organization** (name + slug) — seeds default `owner` and `member` roles
- Open an organization to manage **organization roles** (permission bundles), members, and invites
- **Invite by email** — create a shareable invite link (no email is sent automatically)
- Add an existing tenant user as a member (select an organization role)
- Remove a member from the organization

See [Organizations](/docs/organizations) for how app-scoped orgs relate to directories and user identity.

See [Organization invites](/docs/invites) for the full invite flow and API.

For programmatic org/member management from your BFF, use the [Members API](/docs/api/members) and [Invites API](/docs/api/invites).

## UI development

Admin styles are built from `static/css/admin.input.css` with Tailwind v4:

```bash
cd authstack
make css
```

Static assets are served at `/static`. The Docker image builds CSS in a Node stage before compiling the Rust binary.

## JSON API

Instance admins can still create applications programmatically after logging in:

```bash
curl -c cookies.txt -X POST http://localhost:8080/admin/login \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "email=admin@example.com&password=your-password"

curl -b cookies.txt -X POST http://localhost:8080/admin/applications \
  -H "Content-Type: application/json" \
  -d '{"name": "my-app"}'
```

App admins receive `403 Forbidden` from this JSON endpoint.
