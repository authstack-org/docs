title: Directories & deployment topologies
description: How Authstack scopes identity with directories, grants, and two database schemas.
---

Authstack separates **platform operators** from **tenant identity**. Operators use the admin panel; end users authenticate through your applications. Tenant data is grouped under **directories** — the boundary that decides whether users are isolated per app or shared across apps.

## Database layout

PostgreSQL data is split into two schemas:

| Schema | Purpose | Examples |
| --- | --- | --- |
| **`admin`** | Instance operators and their app access | `admin_user`, `admin_app_grant` |
| **`tenant`** | Directories, applications, users, orgs, grants | `directory`, `application`, `user`, `organization`, `user_app_grant` |

Application code sets `search_path` to `tenant, admin, public` so queries can use unqualified table names. Migrations always run against the default path so migration history stays in `public._sqlx_migrations`.

## Core concepts

```
Instance (single Authstack deployment)
└── Directory                    ← identity boundary
    ├── identity_policy          application_silo | shared_directory
    ├── Applications             client_id + secret per app
    ├── Users                    belong to the directory
    │   └── user_app_grant       which apps a user may access
    └── Organizations            personal or team workspaces
```

- **Directory** — A tenant identity realm (company, customer, or logical partition). Every application belongs to exactly one directory.
- **Identity policy** — Controls how users and organizations relate to applications (see below).
- **Application** — A client that authenticates with `client_id` + `client_secret`. Issues JWTs scoped to one app.
- **User app grant** — Links a user to an application they may sign in to. Created automatically on signup for that app; in shared-directory mode, grants can be added when a user first uses another app.
- **Scoped application** — For `application_silo` users, `scoped_application_id` pins the user to one app (same email can exist as different users in different apps). For `shared_directory`, this is `NULL` — one identity per email in the directory.

Fresh installs seed a **default directory** (`slug: default`, policy `application_silo`). New applications are attached to it automatically unless you pick another directory when creating the app.

Instance admins can create additional directories from the admin panel (**Directories → New directory**) or by inserting into `tenant.directory` directly during development.

## Typical case 1: Startup / Clerk-like (default)

**One company, multiple products, isolated users per app.**

| Setting | Value |
| --- | --- |
| Directories | One (`default`) |
| Policy | `application_silo` |
| Users | Scoped per application; `user@example.com` in App A ≠ same user in App B |
| Organizations | Per application (`application_id` set on team orgs) |
| Grants | One grant per user to the app where they signed up |

This matches most B2B SaaS setups today: each product has its own user table in practice, but you operate one Authstack instance and admin panel.

**Flow:** User signs up via App A → user row with `scoped_application_id = App A` → personal org in App A → JWT includes `app_id` and `directory_id`.

## Typical case 2: Company IdP / enterprise SSO path

**One identity per employee, multiple internal apps.**

| Setting | Value |
| --- | --- |
| Directories | One (or one per business unit later) |
| Policy | `shared_directory` |
| Users | One row per email in the directory (`scoped_application_id` is NULL) |
| Organizations | Directory-wide team orgs (`application_id` is NULL) |
| Grants | Explicit `user_app_grant` per application the user may access |

Same person signs in once conceptually; access to each app is gated by grants. This is the foundation for a future OAuth/OIDC layer where Authstack acts as the identity provider for your suite.

**Flow:** User exists in directory → grant added for App B → user can obtain tokens for App B without a separate signup.

## Typical case 3: Multi-directory hosting (future)

**One Authstack instance, multiple customers or realms.**

| Setting | Value |
| --- | --- |
| Directories | Many (e.g. `acme`, `globex`) |
| Policy | Per directory (`application_silo` or `shared_directory`) |
| Applications | Each app linked to one directory |
| Operators | Instance admins see all; directory admins see scoped operators in their directories; app admins have no access |

Use this when you host Authstack for several companies or environments and need hard identity separation at the directory level. The default install uses a single hidden directory so simple setups never see this complexity.

## JWT claims

Access tokens include:

- `sub` — user id
- `app_id` — application the token was issued for
- `directory_id` — directory that owns the identity
- `org_id` — active organization (optional)

Your BFF should treat `app_id` and `directory_id` as part of the security context when calling Authstack or enforcing authorization.

## Choosing a policy

| If you need… | Use |
| --- | --- |
| Same email as different users in different apps | `application_silo` (default) |
| One employee identity across internal apps | `shared_directory` |
| Minimal concept surface for a new project | Default directory + `application_silo` |
| Path to central SSO later | `shared_directory` + grants |

Changing policy on a live directory is a significant migration; pick `shared_directory` before you have production users if you know you need IdP-style identity.
