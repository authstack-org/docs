title: Directories & deployment topologies
description: How Authstack scopes identity with directories, grants, and two database schemas.
---

Authstack separates **platform operators** from **tenant identity**. Operators use the admin panel; end users authenticate through your applications. Tenant data is grouped under **directories** — the identity boundary for users, organizations, and application access.

## Database layout

PostgreSQL data is split into two schemas:

| Schema | Purpose | Examples |
| --- | --- | --- |
| **`admin`** | Instance operators and their app access | `admin_user`, `admin_app_grant`, `admin_directory_grant` |
| **`tenant`** | Directories, applications, users, orgs, grants | `directory`, `application`, `user`, `organization`, `user_app_grant` |

Application code sets `search_path` to `tenant, admin, public` so queries can use unqualified table names. Migrations always run against the default path so migration history stays in `public._sqlx_migrations`.

## Core concepts

```
Instance (single Authstack deployment)
└── Directory                    ← identity boundary
    ├── Applications             client_id + secret per app
    ├── Users                    one identity per email in the directory
    │   └── user_app_grant       which apps a user may sign in to
    └── Organizations            app-scoped tenants and teams (optional)
```

- **Directory** — A tenant identity realm (company, customer, or logical partition). Every application belongs to exactly one directory. **Users in a directory share one identity** — the same email is one user record for all apps in that directory.
- **Application** — A client that authenticates with `client_id` + `client_secret`. Issues JWTs scoped to one app.
- **User app grant** — Links a user to an application they may sign in to. Created automatically on signup for that app. Users **cannot log in** to an app without a grant, even if they exist in the same directory.
- **Organizations** — App-scoped tenants or teams (not created on signup). Use `user_id` for per-user private data in your app.

### App-level separation

If two applications must **not** share users (different customers, different products with isolated accounts), put them in **separate directories**. There is no per-directory policy toggle — separation is always at the directory level.

If two applications are in the **same directory**, a user who signs up through App A exists as the same identity in that directory. They can only use App B after receiving a `user_app_grant` (via invite, admin provisioning, or another explicit grant path).

Fresh installs seed a **default directory** (`slug: default`). New applications are attached to it automatically unless you pick another directory when creating the app.

Instance admins can create additional directories from the admin panel (**Directories → New directory**).

## Typical case 1: One company, multiple products (SSO-style)

**One directory, several internal apps, shared employee identity.**

| Setting | Value |
| --- | --- |
| Directories | One (`default` or a named directory) |
| Users | One row per email in the directory |
| Access | `user_app_grant` per application |
| Organizations | App-scoped; created explicitly per application |

**Flow:** User signs up via App A → user row in directory → grant for App A → JWT includes `app_id` and `directory_id` (no org until the user joins or creates one). Admin or invite flow adds a grant before the user can sign in to App B.

## Typical case 2: Multi-tenant hosting

**One Authstack instance, multiple customers.**

| Setting | Value |
| --- | --- |
| Directories | One per customer (e.g. `acme`, `globex`) |
| Applications | Each customer's apps in their directory |
| Operators | Instance admins see all; directory admins see scoped directories |

Each customer's users are isolated from other customers because they live in different directories.

## Typical case 3: Single app / startup

**Minimal setup — default directory only.**

| Setting | Value |
| --- | --- |
| Directories | `default` (hidden in simple setups) |
| Applications | One or more in `default` |

Remember: multiple unrelated apps in `default` **share the same user pool**. Use separate directories when apps must not share signups.

## JWT claims

Access tokens include:

- `sub` — user id
- `app_id` — application the token was issued for
- `directory_id` — directory that owns the identity
- `org_id` — active organization (optional)

Your BFF should treat `app_id` and `directory_id` as part of the security context when calling Authstack or enforcing authorization.
